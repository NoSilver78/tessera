## MUSS
1. **allow-only-Assertion am Choke-Point (CRITICAL, §8.1 — fehlt komplett):** `async_set_group_policy` schreibt die Policy **verbatim** (nur Version-/Namespace-/system_generated-Guard) — ein `True`, `{"entities": True}`, `{"domains": …}` oder ein bare-True-Leaf `{"entities":{"entity_ids":{"x": True}}}` würde **ungeprüft persistiert** (genau die Audit-Warnung `spec-e3-enforce.md:46`). **Fix:** fail-closed allow-only-Validator **vor jedem** Gruppen-Write (in `async_set_group_policy` ODER in `apply`s Gruppen-Schleife). Abweisen: Policy ≠ `{"entities":{"entity_ids":{…}}}`-Dict · `entities is True` · jeder `domains`-Key · jedes `entity_ids[x]`-Leaf, das **nicht** `{read/control: bool}` ist (nur `{read,control}`-Leaves erlaubt, `concept.md` C1). Bei Verstoß `UnsafeAuthTarget` + `refused_reason="allow-only"`, **kein Write**. + DoD-Test (e).
2. **Lockout-Precheck (CRITICAL, §6.6 — fehlt):** vor JEDEM Write asserten, dass **≥1 aktiver Owner/Admin** Zugriff behält. Der Helper `RecoveryController.async_assert_no_admin_lockout` existiert (auth_adapter.py:315), wird aber nie gerufen. **Fix:** am Anfang von `apply_enforce_plan` (nach blocked-Check, vor Writes) einen Owner/Admin-Survivor-Check ausführen (über `users_by_id`, ODER `hass`/`RecoveryController` reinreichen) → bei Fehlschlag `refused_reason="lockout"`, kein Write. + DoD-Test (d).
3. **ApplyResult-Contract wiederherstellen (HIGH):** die gelieferte Form (`status`, keine `refused_reason`) macht die geforderte **Refusal-Taxonomie unabbildbar** → DoD-Tests (c)/(d)/(e) unerfüllbar. **Fix:** `refused_reason: "blocked"|"version"|"lockout"|"allow-only"|"write-error"|None` (+ `ok: bool` ODER ein eindeutiges `status`, das diese Fälle trägt) einführen; jede Refusal über diese Taxonomie routen. DI-Signatur darf bleiben — aber die Precheck-Daten (Users/Adapter) müssen verfügbar sein.
4. **`expected_tessera_group_ids` aus AKTUELLER Mitgliedschaft (HIGH — Korrektur MEINES Spec):** Mein E3.3-Task §1 Schritt 4 war **falsch** („expected = tessera:*-Teilmenge von **target**"). Das macht den Adapter-No-Drop-Check (`missing = expected − full`) zur **Tautologie** (immer leer). Korrekt (§8.2): **`expected` = die tessera:*-Teilmenge der AKTUELLEN nativen Gruppen des Users** (`[g for g in _user_group_ids(users_by_id[user_id]) if g.startswith(TESSERA_GROUP_PREFIX)]`) — so fängt `_validate_full_group_superset` einen **unter-berechneten** Target (echter Drop-Schutz). + Test: ein Target, dem eine aktuell gehaltene tessera-Gruppe fehlt → Adapter wirft `IncompleteSuperset`.

## SOLL

Sicherstellen + testen, dass ein Abbruch mid-rebind **keinen Lockout** hinterlässt (noch-nicht-rebound-User behalten ihre alten Gruppen; Orphans erst nach allen Rebinds). Docstring: „Rollback/Two-Phase-Journal = E3.4; bis dahin stop+report, Half-State ist non-lockout + idempotent re-applybar." + Test (Partial-Failure → kein Owner/Admin verliert Zugriff).
